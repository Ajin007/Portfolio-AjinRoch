Section 01

Enterprise AI Software Development Lifecycle

The full delivery pipeline, from business requirements to continuous improvement — with the right Claude capability, an example prompt, expected output, one best practice, and the most common mistake at every stage.

01

Business Requirements

Claude Chat + Projects
How to use Claude
Turn raw stakeholder notes, transcripts, and emails into structured requirement documents with priorities and acceptance criteria.
Best capability
Projects + long context — load all stakeholder inputs into one Project so every draft shares the same source of truth.
Example prompt
"From these 3 meeting transcripts, extract functional + non-functional requirements as a MoSCoW table with open questions."
Expected output
BRD draft, MoSCoW matrix, risk/assumption register, list of unanswered questions for stakeholders.
Best practice
Ask Claude to flag ambiguities and conflicts between stakeholders before writing a single requirement.
Common mistake
Accepting invented details. Instruct: "Only use information present in the sources; mark gaps as TBD."
02

Product Vision

Claude Chat + Artifacts
How to use Claude
Pressure-test the vision: personas, jobs-to-be-done, competitive positioning, and a phased roadmap.
Best capability
Extended thinking + web search for market context; Artifacts for a living one-pager the team iterates on.
Example prompt
"Act as a product strategist. Stress-test this vision doc: list the 5 riskiest assumptions and how to validate each in 2 weeks."
Expected output
Vision one-pager, assumption/validation map, phased roadmap (Now / Next / Later), success metrics.
Best practice
Ask for counter-arguments ("argue why this product fails") — cheaper than a failed quarter.
Common mistake
Using Claude only to polish wording instead of interrogating the strategy itself.
03

Architecture Design

Claude (frontier model) + Artifacts
How to use Claude
Generate candidate architectures, compare trade-offs (CAP, consistency, cost), and produce HLD/LLD documents and diagrams-as-code.
Best capability
Highest-capability model + extended thinking. Architecture errors compound — spend tokens here, not on rework.
Example prompt
"Design a system for 50K concurrent learners taking proctored exams. Give 2 architectures, capacity estimates, failure modes, and a recommendation."
Expected output
HLD with component diagram (Mermaid/SVG), capacity math, ADRs (Architecture Decision Records), rejected alternatives.
Best practice
Always require 2+ options with explicit trade-offs — never accept a single unexamined design.
Common mistake
No constraints given. State budget, team size, SLA, and existing stack, or you get textbook over-engineering.
04

Security Review

Claude + Threat Modeling
How to use Claude
Run STRIDE threat modeling on the HLD, review auth flows, and map controls to SOC2 / ISO 27001 / GDPR obligations.
Best capability
Long context — feed the full HLD + data-flow diagrams so threats are traced end-to-end, not per component.
Example prompt
"STRIDE-analyze this architecture. For each threat: affected component, attack path, likelihood, impact, and mitigating control."
Expected output
Threat model table, prioritized mitigation backlog, compliance gap list, security NFRs added to the design.
Best practice
Security review happens before code exists — a design-stage fix costs 1x, a production fix costs 100x.
Common mistake
Treating Claude's review as sign-off. It accelerates human security engineers; it does not replace them.
05

Data Architecture

Claude + Schema Design
How to use Claude
Design schemas (SQL + NoSQL), choose engines, plan partitioning/sharding, and define the lake → warehouse flow.
Best capability
Skills — a database-engineering skill encodes your naming conventions, indexing rules, and migration standards.
Example prompt
"Model multi-tenant training data: 500 orgs, 2M learners, per-tenant isolation. Compare shared-schema vs schema-per-tenant vs DB-per-tenant."
Expected output
ERD, DDL scripts, index plan, retention/TTL policy, lake/warehouse zone design (raw → curated → serving).
Best practice
Give Claude real query patterns and volumes — schemas are designed for access paths, not for entities.
Common mistake
Designing tables without stating read/write ratios, growth rate, and hot queries.
06

Backend Development

Claude Code
How to use Claude
Agentic implementation in the repo: scaffold services, write endpoints + tests, refactor across files, fix failing builds.
Best capability
Claude Code + CLAUDE.md + MCP — repo context, coding standards, and live access to Jira/GitHub/DB schemas.
Example prompt
"Implement the /enrollments API per openapi.yaml: service layer, validation, unit + integration tests. Run tests and fix failures."
Expected output
Working code on a branch, passing test suite, migration files, updated docs — reviewed as a normal PR.
Best practice
Make Claude verify its own work: run the compiler, run tests, lint — never trust unexecuted code.
Common mistake
Giant vague tasks ("build the backend"). Decompose into PR-sized units with clear acceptance criteria.
07

Frontend Development

Claude Code + Artifacts
How to use Claude
Prototype UI in Artifacts, then implement production React/TS with your design system, state management, and a11y rules.
Best capability
Artifacts for rapid prototypes; Claude Code + a design-system Skill for pixel-consistent production components.
Example prompt
"Build the trainer dashboard per this Figma export using our design-system skill: React Query for data, WCAG AA, empty/loading/error states."
Expected output
Component tree, typed API hooks, Storybook stories, accessibility annotations, responsive behavior.
Best practice
Encode the design system as a Skill once — every screen generated after that stays visually consistent.
Common mistake
Letting each session reinvent styling, producing five subtly different button variants per sprint.
08

Infrastructure

Claude Code + IaC
How to use Claude
Write and review Terraform/Helm, design VPCs and IAM, containerize services, and plan multi-AZ topology.
Best capability
Claude Code against the IaC repo, with a cloud-engineering Skill holding tagging, naming, and cost policies.
Example prompt
"Write Terraform for a 3-AZ EKS cluster with IRSA, private subnets, and least-privilege node roles. Explain every security group rule."
Expected output
Modular Terraform, plan output review, network diagram, cost estimate, runbook for provisioning.
Best practice
Claude drafts, pipelines apply. Never let a model hold cloud credentials or run terraform apply unreviewed.
Common mistake
Copy-pasting generated IaC without a plan/diff review — drift and privilege creep follow.
09

Testing

Claude Code + CI
How to use Claude
Generate unit/integration/E2E tests, derive edge cases from specs, build test data factories, and triage flaky tests.
Best capability
Agentic loops: write test → run → read failure → fix → rerun, until green. Attach coverage reports for gap analysis.
Example prompt
"Coverage is 61% on the billing module. Write tests for uncovered branches, prioritizing money-handling paths. Run and report."
Expected output
New test suites, boundary/negative cases, property-based tests where useful, coverage delta report.
Best practice
Ask for tests that encode the spec, not tests that mirror the implementation (those pass even when the code is wrong).
Common mistake
Chasing 100% coverage of trivial getters while critical business logic branches stay untested.
10

Deployment

Claude + CI/CD Review
How to use Claude
Design pipelines (GitHub Actions/Jenkins), choose rollout strategy (blue-green, canary), and draft rollback runbooks.
Best capability
Claude Code for pipeline YAML; Chat for strategy reviews ("what breaks if the canary metric lags 5 min?").
Example prompt
"Convert this Jenkins pipeline to GitHub Actions with canary deploy to EKS, auto-rollback on p95 latency regression, and required approvals for prod."
Expected output
Pipeline definitions, promotion gates, rollback procedure, release checklist, change-failure-rate baseline.
Best practice
Deployment execution stays deterministic — Claude designs and reviews the pipeline; the pipeline runs it.
Common mistake
Putting an LLM in the hot path of a release. Gates must be scripts and metrics, not model judgment.
11

Monitoring

Claude + Observability MCP
How to use Claude
Define SLOs/SLIs, write alert rules and dashboards, and — via MCP connectors — investigate incidents against live logs and traces.
Best capability
MCP integrations (CloudWatch, Grafana, Sentry): Claude queries real telemetry instead of reasoning from pasted screenshots.
Example prompt
"Error rate on checkout jumped at 14:32. Pull correlated logs, recent deploys, and traces; give a ranked root-cause hypothesis list."
Expected output
SLO definitions, alert rules with thresholds and rationale, incident timeline, root-cause hypotheses with evidence.
Best practice
Alert on symptoms users feel (latency, errors, saturation) and let Claude help trace to causes.
Common mistake
Auto-acting on model hypotheses. Root-cause analysis assists the on-call engineer; humans execute remediation.
12

Optimization

Claude + Profiling Data
How to use Claude
Interpret profiler output, EXPLAIN plans, and flame graphs; propose ranked optimizations with expected impact.
Best capability
File analysis + long context: attach real profiles, slow-query logs, and cost reports — never optimize from vibes.
Example prompt
"Here's the pg_stat_statements top-20 and one EXPLAIN ANALYZE. Rank fixes by impact/effort and write the index migrations."
Expected output
Bottleneck ranking, index/query rewrites, cache strategy, capacity headroom analysis, cloud cost cuts.
Best practice
Measure → change one thing → measure again. Ask Claude to predict the improvement, then verify it.
Common mistake
Applying every suggestion at once — you can't attribute the win, and one regression hides in the batch.
13

Continuous Improvement

Claude Routines + Projects
How to use Claude
Run retro synthesis, DORA-metric trend analysis, incident postmortem drafting, and knowledge-base upkeep on a schedule.
Best capability
Scheduled Routines — weekly engineering-health reports and postmortem digests generated automatically.
Example prompt
"Every Friday: summarize merged PRs, incidents, and DORA metrics for the week; flag trends and propose one process experiment."
Expected output
Weekly health report, postmortems with action items, updated runbooks, a living lessons-learned library.
Best practice
Feed improvements back into Skills and CLAUDE.md — the org's Claude gets smarter with every retro.
Common mistake
Generating reports nobody owns. Every automated insight needs a named human who acts on it.

Section 03

Load Optimization Framework

Six optimization domains, from the Linux kernel up to Kubernetes autoscaling — and the four ways Claude accelerates each of them.

Compute: CPU · Memory · Kernel

  • CPU profiling, hot-path analysis, vectorization
  • Memory: heap sizing, GC tuning, leak hunting
  • Linux kernel: scheduler, cgroups, huge pages, ulimits, TCP stack tuning

Claude: reads flame graphs and GC logs, explains kernel parameters, proposes sysctl changes with risk notes.

I/O & Network

  • Disk: IOPS budgets, io_uring, write batching
  • Network: connection pooling, keep-alive, payload compression, gRPC vs REST
  • Latency budgets per hop

Claude: traces a request's latency budget end-to-end and identifies the dominant hop from real trace data.

Caching: Redis · CDN

  • Cache-aside vs write-through vs write-behind
  • TTL and invalidation strategy, stampede protection
  • CDN edge caching, cache-key design

Claude: designs the cache hierarchy, computes hit-rate targets, writes invalidation logic and stampede guards.

Concurrency: Async · Thread Pools

  • Async processing for non-blocking workloads
  • Thread-pool sizing (Little's Law), backpressure
  • Queue-based load leveling

Claude: sizes pools from arrival rate and service time, spots blocking calls inside async paths in code review.

Scaling: Horizontal · Vertical · Auto

  • Horizontal: stateless services, sticky-session removal
  • Vertical: right-sizing before scaling out
  • Kubernetes: HPA/VPA, PodDisruptionBudgets, cluster autoscaler

Claude: writes HPA policies tied to real metrics, reviews readiness probes, plans safe scale-in behavior.

Database Optimization

  • Index design, query rewriting, plan analysis
  • Connection pooling (PgBouncer), read replicas
  • Partitioning and archival of cold data

Claude: turns EXPLAIN ANALYZE output into ranked fixes and writes the exact index migration scripts.

Performance Analysis

Feed Claude load-test results, APM exports, and p95/p99 trends; it produces a structured performance assessment against your SLOs.

Bottleneck Detection

Claude correlates metrics across layers (CPU flat, latency up, queue depth rising → downstream dependency) and names the constraint.

Profiling Interpretation

Flame graphs, heap dumps, slow-query logs, and kernel perf output translated into plain-language findings engineers act on.

Optimization Recommendations

Every recommendation ranked by expected impact vs effort vs risk — with the measurement plan to verify the win afterward.

Section 06

Automation Framework

The automation ladder — every rung from shell scripts to Claude Routines — and the decision tree that keeps judgment tasks with Claude and deterministic tasks in pipelines.

A1

Routine Tasks → Scripts

Deterministic
What lives here
Fixed-logic, repeatable work: backups, log rotation, data syncs, report exports. Claude writes the scripts; cron/systemd runs them.
Rule
If the steps never change, the execution should never involve a model.
A2

Claude Routines → Workflows

Scheduled AI
What lives here
Scheduled tasks needing language understanding: weekly team reports, meeting-transcript → task extraction, inbox triage, digest generation.
Rule
Same schedule + same shape of output, but the content requires reading and judgment → Routine.
A3

CI/CD → GitHub Actions · Jenkins

Deterministic core
What lives here
Build, test, scan, deploy gates. Claude authors and reviews pipeline YAML and can be invoked as a step (PR review comments, changelog drafts) — but pass/fail gates stay scripted.
Rule
Model output can inform a gate; it must never be the gate for production.
A4

Infrastructure · Testing · Release · Docs · Monitoring Automation

Terraform + AI authoring
Infrastructure
Terraform modules drafted by Claude, applied only through plan-reviewed pipelines.
Testing & Release
Claude generates and repairs tests; release trains, versioning, and rollback stay deterministic.
Docs & Monitoring
Docs regenerated from code on merge; alert rules and dashboards maintained as code with Claude proposing diffs.
Does the task require reading, judgment, or synthesis of unstructured input?
Yes → use Claude
Complex + interactive → Claude Chat / Claude Code session
Recurring on a schedule → Claude Routine
Multi-system, multi-step → Agent with MCP tools + human checkpoint
No → keep it deterministic
Fixed logic, any frequency → script / cron
Build-test-deploy → CI/CD pipeline
Infra state → Terraform, reviewed plans only

Never delegate to a model: production deploy approval, secrets handling, irreversible data deletion, financial transactions, compliance sign-off. Always automate: anything a human does identically more than twice a week.

Claude Agent Architecture

A multi-agent delivery system: an orchestrating Planner fans work out to specialist agents, a Reviewer validates everything, and shared context flows through a common workspace. Every agent's contract — responsibility, inputs, outputs — is explicit.

AgentResponsibilityInputsOutputs
PlannerDecompose the request into tasks, order dependencies, assign to specialists, track completion.User goal, constraints, prior plan stateplan.md — task graph with owners and done-criteria

Backend / Frontend / Database / DevOpsImplement within their boundary, self-verify (compile, test, lint) before hand-off.Contracts + relevant repo slice onlyBranches with passing checks, migration files, IaC diffs

ReviewerValidate all work against the original spec; reject with reasons; assemble the final hand-off.Everything on the context busApproval / rejection with actionable feedback

Context sharing

Agents communicate through files, not chat: plan.md, specs, and reports in a shared workspace. Each agent receives only its relevant slice — full-context broadcast degrades every agent at once.

Memory

Long-lived knowledge (conventions, past decisions, gotchas) lives in CLAUDE.md and Skills; task state lives in the plan file. Agents are stateless between runs — the workspace is the memory.

Coordination & validation

Planner owns sequencing; Reviewer owns quality; Security owns veto. Every hand-off is validated against explicit done-criteria — "compiles and tests pass" is the floor, not the bar.

Section 09

Claude Context Engineering

Context is an architecture, not a paste-bin. Twelve layers, ordered from most stable (bottom of your setup, configured once) to most volatile (the request of the moment).

1 · User Request

Volatile. The specific ask. Should be short — because everything below already carries the weight.

2 · Business Context

Domain, customers, constraints, what "good" means commercially. Lives in the Project description.

3 · Project Context

Goals, milestones, current sprint, known issues. Lives in Project knowledge, refreshed weekly.

4 · Repository Context

Code structure, key modules, build commands. Claude Code discovers this; CLAUDE.md pins the essentials.

5 · Architecture Context

ADRs, service boundaries, contracts. Prevents locally-correct, globally-wrong changes.

6 · Coding Standards

Naming, patterns, test requirements, lint rules — in CLAUDE.md so every session obeys them without being told.

7 · Security Rules

Forbidden patterns, secret-handling policy, data classification. Non-negotiable, stated explicitly.

8 · Company Guidelines

Tone, compliance requirements, approval workflows, brand rules for anything client-facing.

9 · Memory

Accumulated knowledge of you and past work — corrections made once should never be needed twice.

10 · Examples

Two or three gold-standard samples of the desired output beat a page of description.

11 · Skills

Packaged expertise loaded on demand — procedures, references, and templates that don't consume context until needed.

12 · Prompt → Claude

The assembled context window. Principle: stable knowledge in configuration, volatile knowledge in the message. Curate ruthlessly — irrelevant context actively degrades output.

  • ADR template + approved patterns

  • Error/response envelope standard

Frontend Skill

  • Design-system tokens + components
  • State-management rules
  • Accessibility checklist (WCAG AA)

Security Skill

Activates: auth code, PRs touching data, IAM.

  • Secure-coding ruleset
  • Secret-handling policy
  • Threat-model template

DevOps Skill

Activates: pipelines, Docker, Terraform, K8s.

  • IaC module standards + tagging
  • Pipeline stage requirements
  • Rollout/rollback playbooks

Testing Skill

Activates: test writing or coverage work.

  • Test pyramid ratios per repo

Documentation Skill

  • Doc templates per artifact type

  • Naming + indexing standards

Section 11

UI/UX Design Workflow

Ten stages from requirements to optimized production UI. Orange marks the design activity; the purple line states Claude's role at each step.

01 · Requirements

User research synthesis, JTBD, success metrics.

Claude: clusters interview transcripts into themes; drafts personas grounded only in the evidence provided.

02 · Wireframes

Low-fidelity structure, content hierarchy, layout options.

Claude: generates 2–3 layout concepts as HTML wireframes in Artifacts for instant comparison.

03 · User Flow

Task flows, decision points, error and empty states.

Claude: maps every path including the unhappy ones — the flows designers forget are the ones users find.

04 · Component Design

Reusable components, variants, interaction states.

Claude: defines component APIs (props, states, events) and drafts each variant against the token set.

05 · Design System

Tokens, typography scale, spacing, color semantics.

Claude: codifies the system as a Skill so every future screen inherits it automatically.

06 · Accessibility

WCAG AA, keyboard paths, contrast, screen readers.

Claude: audits markup for ARIA/contrast/focus issues and fixes them — accessibility as a build step, not an afterthought.

07 · Prototype

Interactive prototype for stakeholder validation.

Claude: builds clickable HTML/React prototypes in Artifacts in minutes — test the idea before committing sprints.

08 · Frontend Code

Production implementation with real data wiring.

Claude Code: converts the validated prototype into production components with types, tests, and stories.

09 · Testing

Visual regression, interaction tests, usability findings.

Claude: writes Playwright interaction tests and turns usability session notes into a prioritized fix list.

10 · Optimization

Core Web Vitals, bundle size, perceived performance.

Claude: reads Lighthouse traces, proposes code-splitting and image strategy, verifies the vitals afterward.
