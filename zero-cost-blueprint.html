Section 01

Rules of the Free Game

"Free" is the most overloaded word in developer marketing. Learn the four species of free before betting your project on any of them.

Truly free

No card, no expiry, no meaningful cap for your scale. Examples: GitHub public repos, Cloudflare Pages, open-source software, government open data, GitHub Pages.

Free tier

Ongoing but capped: requests/day, storage, cold starts, sleep-after-idle. Fine if you architect around the cap. Examples: Render, Neon, Supabase, Gemini API.

Free trial

Credits or a clock that runs out — then bills or breaks. Use only for evaluation, never as a foundation. Examples (2026): Railway credits, most cloud "$300 credit" offers.

Free because you host it

Open source on hardware you already own — your laptop, an old PC, your 32GB Ollama box. Zero rent forever; you pay in setup time. Often the strongest option.

Before adopting anything "free", answer four questions:
1 · Card required?
If a credit card is needed up front, treat it as a paid product on a delay. Prefer no-card options.
2 · What's the real cap?
Requests/day, tokens/day, build minutes, storage, MAU. The headline number is rarely the binding one.
3 · Commercial use OK?
Many free tiers are personal-use only. If a client or revenue is involved, read the terms first.
4 · Exit path?
Standard tech (Postgres, Docker, Git) migrates anywhere. Proprietary lock-in on a free tier is a debt.

Long context / PDFs

Bulk / overnight batch

Each provider's limits are independent — get all the keys on day one, put them behind one OpenAI-compatible client (they nearly all support the same SDK), and switch base_url per task. That's a production-grade AI layer at $0/month.

Section 07 · Addon

Build Your Own API — The Provider Blueprint

Flip sides of the table: instead of only consuming free APIs, ship one. Ten stages from idea to a public API others can use — every stage at $0, with the monetization path left open for later.

P1

Decide the product

Design
What makes a good API
Data or capability that is hard to assemble but easy to consume: a cleaned dataset, a calculation engine (your OI-pulse signals!), a converter, a domain lookup (Indian exam calendars, Java cert question metadata).
Validate demand free
Search the public-apis repo and RapidAPI — if the niche is empty or the existing options are bad, that's your gap.
Rule
One resource, done excellently, beats ten endpoints done vaguely. v1 should be embarrassingly small.
P2

Design the contract first

OpenAPI
REST conventions
Nouns not verbs (GET /v1/signals, not /getSignals), plural resources, standard status codes, pagination from day one (?limit=&cursor=).
Error envelope
One consistent error shape: {"error":{"code":"RATE_LIMITED","message":"…"}}. Consumers integrate against your errors as much as your data.
Version in the URL
/v1/ from the first release. You can never remove a field from v1; you can always ship v2.
P3

Build it — FastAPI skeleton

Free stack
Minimal production-shaped API: key auth + rate limit + auto docs (all free, ~40 lines)
from fastapi import FastAPI, Depends, HTTPException, Request from fastapi.security import APIKeyHeader from slowapi import Limiter from slowapi.util import get_remote_address limiter = Limiter(key_func=get_remote_address) app = FastAPI(title="OI Pulse API", version="1.0.0", description="Options OI signal analytics") # /docs is auto-generated api_key_header = APIKeyHeader(name="X-API-Key") KEYS = {"demo_key_123": {"tier": "free", "rpd": 100}} # later: move to DB def auth(key: str = Depends(api_key_header)): if key not in KEYS: raise HTTPException(401, {"error": {"code": "INVALID_KEY", "message": "Get a free key at /signup"}}) return KEYS[key] @app.get("/v1/signals") @limiter.limit("10/minute") # free-tier rate limit def signals(request: Request, symbol: str, user=Depends(auth)): return {"symbol": symbol, "signal": "bullish", "confidence": 0.72, "as_of": "2026-07-05"} @app.get("/v1/health") # uptime monitors hit this def health(): return {"status": "ok"}
Stack (all free)
FastAPI (or Express/Hono for JS) · slowapi for rate limiting · SQLite/Neon for keys & data · pytest + httpx for tests · Swagger UI ships automatically at /docs.
P4

Secure it

Non-negotiable
API keys
Issue per-consumer keys (prefix them: oip_live_…). Store hashes, never plaintext. Self-serve signup keeps support at zero.
Rate limiting per tier
Free: e.g. 100 req/day, 10/min. Return 429 + Retry-After header. This is what makes a free tier survivable.
Also
HTTPS only (free via Cloudflare/host), CORS allow-list, input validation on every param (FastAPI's Pydantic does it), never leak stack traces in errors.
P5

Document like a product

Adoption = docs
Auto reference
FastAPI's built-in Swagger (/docs) + ReDoc (/redoc) give an interactive reference for free.
Quickstart page
A single page: get a key → one curl command → real response. If a stranger can't get a 200 in 5 minutes, the docs failed.
Host free
Docs site on GitHub Pages or Cloudflare Pages with MkDocs Material (free, beautiful, searchable).
P6

Deploy at $0

Ship
Light & fast API
Cloudflare Workers (rewrite in Hono/JS): 100K req/day free, no cold starts, global edge — the best free home for a public API.
Python / heavier logic
Render free (accept the cold start) or your own box + Cloudflare Tunnel for zero limits. CI/CD: GitHub Actions deploys on push to main.
Data updates
A scheduled GitHub Action (free cron) refreshes your dataset nightly and commits/pushes it — a zero-cost ETL pipeline.
P7

Operate

Trust
Uptime
UptimeRobot free pings /v1/health every 5 min (and conveniently keeps Render awake). Free public status page included.
Errors & logs
Sentry free (5K errors/mo) for exceptions; structured JSON logs; provider dashboards for traffic.
Changelog
A CHANGELOG.md and a deprecation policy ("6 months notice") — this is what separates an API people build on from a toy.
P8

Distribute

Get consumers
Free channels
Open a PR adding your API to the public-apis repo · list on RapidAPI (free listing, they handle keys/billing) · post the quickstart on dev.to / Reddit / LinkedIn.
Give SDK-level ease
A copy-paste snippet for Python, JS, and curl in the README. Publishing a tiny pip/npm client package (free) multiplies adoption.
Teach with it
Your own API becomes training material — batches learn REST by consuming something their trainer actually ships.
P9

Monetize later (optional)

Future
The standard pattern
Keep a generous free tier forever (it's your marketing) + paid tiers for volume/commercial use — exactly the model every API in Section 05 uses on you.
Zero-cost billing
List paid tiers through RapidAPI's marketplace — they handle subscriptions and payouts so you never build billing.
Design now, charge later
Because keys + tiers + rate limits exist from P4, flipping a tier from free to paid is a config change, not a rewrite.
Section 08

Golden Rules & Traps

The five golden rules

  • No credit card, ever, for "free". If it asks for a card, it's a paid product waiting to happen.
  • Stack independent free tiers. Limits are per-provider — combining Gemini + Groq + Cerebras + Ollama is a real AI budget.
  • Build on standards (Postgres, Docker, OpenAPI, Git) so any free tier can be swapped in an afternoon.
  • Public repos unlock the best free tiers — GitHub Actions, Pages, and most OSS-friendly perks are richest on public code.
  • Self-hosting is the ultimate free tier. Hardware you already own has no rate limits and no terms of service.

The five traps

  • "Free trial" ≠ "free tier." Credits expire; the clock starts on day one.
  • Auto-pause outages: Supabase pauses after 7 idle days; Render sleeps — schedule a free keep-alive ping if users depend on it.
  • Non-commercial clauses: Vercel Hobby, many data APIs, ip-api free — the moment money is involved, re-read terms.
  • The headline limit isn't the binding one: tokens/day often bind before requests/day; build minutes before bandwidth.
  • Free tiers change monthly. Everything here verified July 2026 — re-check the pricing page before you depend on it.
