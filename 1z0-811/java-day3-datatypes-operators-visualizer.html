Integer literals default to int; decimal literals default to double. That's why float f = 3.14; fails ("possible lossy conversion") but float f = 3.14f; compiles.
Defaults apply only to fields (instance/static variables). Local variables have NO default — reading an uninitialised local is a compile error: "variable x might not have been initialized".
WRONG: "char is 8-bit like C."  →  RIGHT: Java char is 16-bit unsigned Unicode — it can hold 'அ' or '中'.

📍 Variables: local · instance · static · final

public class Counter {
    static int total = 0;          // STATIC — one copy per CLASS, shared by all objects
    int count;                     // INSTANCE — one copy per OBJECT, default 0
    final int LIMIT = 100;         // FINAL — assign once, never reassign

    void increment() {
        int step = 1;              // LOCAL — lives only during this method call, NO default
        count += step; total += step;
    }
}

Small → large happens automatically — no data loss risk, no cast needed.

char → int → long → float → double (char joins at int; char ↔ short never auto-convert — both 16-bit but short is signed).

int i = 100;
long l = i;      // ✅ int → long automatic
double d = l;    // ✅ long → double automatic (may lose PRECISION, not magnitude)
char c = 'A';
int code = c;    // ✅ char → int gives 65
long → float is widening (64-bit → 32-bit!) because float's range is larger even though precision is smaller. Classic exam curveball.
Section 4

↘️ Narrowing (Explicit Cast)

Large → small requires an explicit cast — you accept responsibility for data loss.

double d = 9.99;
int i = (int) d;          // i = 9  → decimal TRUNCATED, not rounded

int big = 130;
byte b = (byte) big;      // b = -126 → OVERFLOW wraps around (130-256)

byte x = 10, y = 20;
byte sum = (byte)(x + y);  // cast REQUIRED: x+y is promoted to int

byte z = 10;
z += 5;                    // ✅ compound assignment has a HIDDEN implicit cast
CaseResult
byte b = 100;✅ int literal fits in byte → implicit narrowing allowed for compile-time constants
byte b = 128;❌ compile error — constant out of range
int n=100; byte b = n;❌ compile error — variable (not constant) needs cast even if value fits
double → intTruncates toward zero: (int)-9.9 = -9
Two guaranteed exam patterns: (1) byte + byte = int — any arithmetic on byte/short/char promotes to int first. (2) b += 5 compiles where b = b + 5 doesn't, because compound operators auto-cast.
Section 5

➗ Arithmetic, Relational & Assignment Operators

Arithmetic: + - * / %

int a = 7 / 2;        // 3   — int/int = int (truncated)
double d = 7 / 2;     // 3.0 — division ALREADY happened in int!
double e = 7.0 / 2;   // 3.5 — one operand double promotes the operation
int r = 7 % 2;        // 1   — remainder; -7 % 2 = -1 (sign follows DIVIDEND)
int boom = 7 / 0;     // 💥 ArithmeticException at RUNTIME
double inf = 7.0 / 0; // Infinity — floating division by zero does NOT throw

Relational: < > <= >= == != → always boolean

Assignment: = += -= *= /= %=

Integer division sneaks into averages: (a+b)/2 with ints drops the .5. Fix by making one operand double: (a+b)/2.0.
WRONG: "% works only on ints."  →  RIGHT: 7.5 % 2 = 1.5 — modulo works on floating-point too.
Section 6

⏫ Increment & Decrement (++ / --)

FormMeaningExpression value
++x (prefix)Increment FIRST, then useNew value
x++ (postfix)Use FIRST, then incrementOld value
int x = 5;
System.out.println(x++);   // prints 5, x is now 6
System.out.println(++x);   // prints 7

int a = 3;
int b = a++ + ++a;          // 3 + 5 = 8   (a: 3→4 after a++, 4→5 before ++a)

int i = 1;
i = i++;                    // i stays 1! old value (1) assigned back over the increment
Trace left-to-right and update the variable immediately at each ++/--. The i = i++ pattern (result: unchanged) is a certification classic.

String s = null;
if (s != null && s.length() > 0) { }   // ✅ safe — && skips length() when s is null

// Ternary: condition ? valueIfTrue : valueIfFalse
int max = (a > b) ? a : b;
String grade = marks >= 90 ? "A" : marks >= 75 ? "B" : "C"; // nested — right-assoc
Side-effects inside skipped operands never run: in if(false && x++ > 0), x is not incremented. Exams love counting increments around short-circuits.

System.out.println("Sum: " + 1 + 2);   // "Sum: 12" — L→R string concat!

Mnemonic ladder: Unary → Arithmetic (*,/,% then +,-) → Relational → Equality → && → || → ?: → =. When in doubt in real code, add parentheses — they cost nothing.

Integer boxed = 10;        // AUTOBOXING: int → Integer (compiler inserts valueOf)
int back = boxed;          // UNBOXING: Integer → int (compiler inserts intValue)

Integer a = 127, b = 127;
System.out.println(a == b);        // true  — Integer CACHE covers -128..127
Integer c = 128, d = 128;
System.out.println(c == d);        // false — outside cache → different objects
System.out.println(c.equals(d));   // true  — ALWAYS compare wrappers with equals()

Integer n = null;
int x = n;                          // 💥 NullPointerException during unboxing
Why wrappers exist: collections hold only objects — ArrayList<Integer>, never ArrayList<int>. Autoboxing bridges the gap invisibly.
The -128 to 127 cache + null-unboxing NPE are the two most-tested wrapper facts in Oracle exams.
