🐞 Error Types: Syntax · Logical · Runtime

TypeDetectedExampleWho catches it
Syntax (compile-time)By javac — nothing runs until fixedMissing ; · unmatched braces · b = b + 1 on byte · undeclared variableCompiler
Runtime (exception)By the JVM during execution÷ by zero · null access · bad array index · bad user inputtry-catch or program crashes
LogicalNever automatically — program runs, answer is wrongUsing i<=n instead of i<n · wrong formula · == instead of equalsTesting & debugging only
Certification wording: "fails to compile" → syntax; "throws an exception at runtime" → runtime; "compiles and runs but prints the wrong value" → logical. Every code question resolves to one of these three verdicts.

🌳 The Throwable Hierarchy

Throwable ├── Error — JVM-level, DON'T catch (unchecked) │ ├── OutOfMemoryError │ └── StackOverflowError — e.g. runaway recursion └── Exception — application-level problems ├── IOException — CHECKED: compiler forces handle-or-declare ├── SQLException — CHECKED └── RuntimeException — UNCHECKED: compiler stays silent ├── NullPointerException ├── ArithmeticException ├── ArrayIndexOutOfBoundsException ├── NumberFormatException └── ClassCastException
CheckedUnchecked
ExtendsException (but NOT RuntimeException)RuntimeException or Error
Compiler forces handling?✅ try-catch or throws clause❌ optional
RepresentsRecoverable external failures (file missing, network down)Programming bugs (null, bad index) / JVM disasters
ExamplesIOException, SQLException, InterruptedExceptionNPE, ArithmeticException, OutOfMemoryError
WRONG: "Errors and exceptions are the same."  →  RIGHT: Both extend Throwable, but Error = JVM catastrophe (don't catch), Exception = application problem (handle it).
Section 3

🛡️ try-catch Block

try {
    int[] a = new int[3];
    a[5] = 10;                                    // 💥 throws here…
    System.out.println("never reached");          // …rest of try is SKIPPED
} catch (ArrayIndexOutOfBoundsException e) {      // most SPECIFIC first
    System.out.println("Bad index: " + e.getMessage());
} catch (Exception e) {                            // broader safety net LAST
    System.out.println("Something else");
}
System.out.println("Program continues");           // ✅ recovery achieved
RuleDetail
Catch orderChild before parent. catch(Exception) before catch(ArithmeticException) = compile error: "exception has already been caught" (unreachable).
Multi-catch (Java 7+)catch (IOException | SQLException e) — one block, unrelated types only (no parent-child pairs), e is implicitly final.
Flow on throwRemaining try statements skipped → first matching catch runs → execution continues AFTER the try-catch.
No matchException propagates up the call stack; if main can't handle it, JVM prints the stack trace and the program dies.
try cannot stand alone — it needs at least one catch or finally. A bare try { } is a compile error.
Section 4

🔚 finally Block

try {
    int r = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Caught");
} finally {
    System.out.println("Cleanup — ALWAYS runs");   // exception or not, return or not
}
ScenarioDoes finally run?
No exception thrown✅
Exception thrown and caught✅
Exception thrown, NOT caught✅ (runs before propagation)
return inside try or catch✅ finally runs BEFORE the method actually returns
System.exit(0) in try❌ the only common escape — JVM halts
// try-with-resources (Java 7+): auto-close, no manual finally needed
try (Scanner sc = new Scanner(System.in)) {
    // sc.close() is called automatically — resource must implement AutoCloseable
}
A return in finally overrides any return/exception from try or catch — it even swallows exceptions. Legal, terrible practice, and a favourite exam puzzle: "which value is returned?" → always the finally one.

🔍 Debugging & Reading Stack Traces

Exception in thread "main" java.lang.ArithmeticException: / by zero
    at Calculator.divide(Calculator.java:12)      ← WHERE it exploded (read TOP line first)
    at Calculator.process(Calculator.java:7)      ← which method called divide
    at Calculator.main(Calculator.java:3)         ← the call chain back to main
PartTells you
Line 1Exception TYPE + message ("/ by zero") — the what
First "at" lineExact class, method, file and line number of the throw — the where. Go here first.
Following "at" linesThe call chain (newest → oldest) — the how it got there
"Caused by:"A wrapped original exception — the true root cause hides at the BOTTOM-most Caused by

Section 7

✅ Exception Best Practices

DoDon't
Catch the most specific type you can handlecatch (Exception e) {} as a reflex
Log or act in every catch (at least e.printStackTrace() in class)Empty catch blocks — errors vanish silently, debugging becomes archaeology
Validate first (if != null, hasNextInt) — prevention over cureUsing exceptions for ordinary flow control (slow + unreadable)
Release resources in finally / try-with-resourcesLeaking Scanners, files, connections
Throw IllegalArgumentException for bad params in your own methodsReturning magic values like -999 to signal errors
// Validation-first pattern:
if (sc.hasNextInt()) {
    int n = sc.nextInt();
} else {
    System.out.println("Please enter a number");
    sc.next();   // discard the bad token or the loop spins forever
}
An empty catch compiles perfectly — exams may show one and ask for output: the program continues normally after the swallowed exception, often printing a misleading "success" line.
