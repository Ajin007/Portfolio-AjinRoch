Section 1

🏛️ Classes & Objects

public class Employee {              // CLASS = blueprint
    String name;                      // state (fields)
    double salary;

    void work() {                     // behaviour (methods)
        System.out.println(name + " is working");
    }
}

Employee e1 = new Employee();        // OBJECT = instance created on the HEAP
e1.name = "Kaafar";
Employee e2 = e1;                     // copies the REFERENCE — both point to ONE object
e2.name = "Kamachi";
System.out.println(e1.name);         // "Kamachi" — same object!
ConceptDetail
ClassCompile-time blueprint: fields + methods + constructors. Zero memory for data until instantiated.
ObjectRuntime instance on the heap; each has its own copy of instance fields.
ReferenceThe variable holds an address, not the object. = copies addresses; null = points nowhere.
new does three things: allocates heap memory → runs the constructor → returns the reference. Losing all references makes the object eligible for garbage collection.

RuleDetail
Top-level classesOnly public or default — a private/protected top-level class is a compile error (private applies to nested classes only).
"default" ≠ keywordPackage-private is the ABSENCE of a modifier. (The word default belongs to switch/interfaces.)
ConventionFields private; methods public as needed; helpers private.
Memory ladder from tightest to widest: private → default → protected → public. "Same package OR subclass" = protected; "same package only" = default.

📍 Instance vs Static vs Local Variables

public class Student {
    static String college = "Parul University"; // STATIC — ONE copy shared by all
    String name;                                 // INSTANCE — one per object

    void show() {
        int year = 2026;                         // LOCAL — this call only, no default
        System.out.println(name + " @ " + college + " " + year);
    }
}

Student a = new Student(); a.name = "Ravi";

Student.college = "PU Vadodara";   // changes for BOTH a and b — shared!

Section 4

🏗️ Constructors

public class Employee {
    String name; double salary;

    Employee() {                             // no-arg constructor
        this("Unassigned", 0.0);             // this() = constructor CHAINING — must be line 1
    }
    Employee(String name, double salary) {  // parameterized
        this.name = name;
        this.salary = salary;
    }
}
RuleDetail
NameExactly the class name — case-sensitive
Return typeNONE — not even void. Adding void makes it an ordinary (never auto-called) method — evil exam trick
Default constructorCompiler inserts a no-arg constructor ONLY when you wrote zero constructors
The classic trapWrite only Employee(String n) → new Employee() is now a compile error (default is gone)
this(…)Calls a sibling constructor; must be the FIRST statement; no two constructors may call each other (cycle = error)
Can be privateYes — utility classes (Math) and singletons block outside instantiation
WRONG: "Constructors are inherited."  →  RIGHT: They are never inherited; a subclass declares its own and chains up with super(…).
Section 5

🎭 Constructor Overloading

public class Book {
    String title; String author; double price;

    Book()                                        { this("Untitled", "Unknown", 0.0); }
    Book(String t)                                { this(t, "Unknown", 0.0); }
    Book(String t, String a)                     { this(t, a, 0.0); }
    Book(String t, String a, double p)           { title=t; author=a; price=p; }
}
// Caller picks the convenient form:
new Book();  new Book("OCP 17");  new Book("OCP 17", "Selvan", 999.0);

Overloaded constructors differ in the parameter list (count, types, or order) — the "telescoping" pattern above funnels every form into one master constructor so initialisation logic lives in exactly one place.

Two constructors whose parameter lists differ only by parameter names or only by return… wait, constructors have no return — so Book(String title) vs Book(String author) = duplicate, compile error. Types must differ, not names.

📞 Methods & Method Calling

//  modifier  return  name      parameters
public   double  applyBonus(double salary, double pct) {
    return salary + salary * pct / 100;    // return ends the method immediately
}

double updated = emp.applyBonus(50000, 10);   // arguments matched by position
ConceptDetail
SignatureMethod name + parameter types. (Return type is NOT part of the signature.)
voidReturns nothing; a bare return; may still exit early.
Parameters vs argumentsParameters = declaration placeholders; arguments = actual values at the call.
Pass-by-value — ALWAYSPrimitives: a copy of the value (caller unaffected). References: a copy of the reference — the method can mutate the shared object, but reassigning the parameter doesn't rebind the caller's variable.
void change(int x, int[] arr) {
    x = 99;          // caller's int unchanged
    arr[0] = 99;     // caller's array IS changed (shared object)
    arr = new int[5];// caller's reference NOT rebound
}
"Java is pass-by-value, and for objects the value passed is the reference" — memorise that sentence; it settles every exam variant of this question.
Section 7

🎯 Method Overloading

class Printer {
    void show(int x)            { System.out.println("int"); }
    void show(double x)         { System.out.println("double"); }
    void show(String x)         { System.out.println("String"); }
    void show(int x, int y)     { System.out.println("two ints"); }
}
p.show(5);        // "int"    — exact match wins
p.show(5L);       // "double" — no long version: widen long→double
p.show('A');      // "int"    — char widens to int before boxing is tried
RuleDetail
What may differParameter count, types, or order
What is NOT enoughReturn type alone, parameter names alone, access modifier — all give "method already defined"
Resolution orderExact match → widening → autoboxing → varargs (compiler stops at first success)
Decided whenCompile time — hence "compile-time / static polymorphism"
show(null) with both show(String) and show(Object) overloads calls the most specific (String). With two unrelated types (String, Integer) it's ambiguous → compile error.
Section 8

🔒 Getters, Setters & this

public class Student {
    private String name;              // 1. private field
    private int age;

    public String getName() {         // 2. public getter
        return name;
    }
    public void setName(String name) {// 3. public setter
        this.name = name;              // this. distinguishes field from parameter
    }
    public void setAge(int age) {
        if (age > 0 && age < 120)      // 4. VALIDATION — the real payoff
            this.age = age;
        else
            throw new IllegalArgumentException("Invalid age: " + age);
    }
}
Convention (JavaBean)Pattern
GettergetXxx() — booleans use isXxx()
SettersetXxx(value), returns void
Read-only propertyGetter without setter (+ often final field)
Why not public fields? Setters give you a checkpoint: validation, logging, computed updates — all impossible with direct field access. Frameworks (Spring, JPA, JSON mappers) also discover properties via these exact naming patterns.
Section 9

⚡ Static Methods

public class MathUtil {
    static int counter = 0;

    public static int square(int n) {   // belongs to the CLASS, no object needed
        return n * n;
    }
    public void instanceThing() { }
}
MathUtil.square(5);                      // call via class name — the whole point
RuleDetail
Static → instance❌ A static method cannot touch instance fields/methods directly, and cannot use this — there IS no object. ("non-static … cannot be referenced from a static context" — the error every beginner meets in main.)
Instance → static✅ Instance methods may freely call static members.
Why main is staticJVM must call it before any object exists.
Via a referenceobj.staticMethod() compiles but is resolved by the reference type at compile time — misleading; always use the class name.
Use forUtilities that need no per-object state: Math.pow, Integer.parseInt, Arrays.sort, factory helpers.
Fixing the main-method error the wrong way: marking every field static "because it compiles" destroys per-object state. Right fix: create an object in main and call instance methods on it.

🖨️ toString()

public class Employee {
    String name; double salary;

    @Override                                    // compiler checks you really override
    public String toString() {
        return "Employee{name='" + name + "', salary=" + salary + "}";
    }
}

Employee e = new Employee();
System.out.println(e);        // println calls toString() automatically
String s = "Emp: " + e;        // so does string concatenation
FactDetail
Default (Object)ClassName@hexHashCode like Employee@1b6d3586 — useless for humans
Signature must be exactpublic String toString() — no args. toString(int x) is an overload, not an override; @Override would flag it.
Auto-invoked byprintln/print, string concatenation, String.valueOf, logger frameworks
Records (Java 16+) auto-generate toString/equals/hashCode — mention as the modern shortcut; classic classes still dominate exam questions.

Reference skeleton

public class Employee {
    private static int count = 0;
    private final String id;
    private String name;
    private double salary;

    public Employee() { this("Unassigned", 0); }
    public Employee(String name, double salary) {
        count++;
        this.id = "EMP" + count;
        this.name = name;
        setSalary(salary);                       // reuse validation
    }
    public void setSalary(double s) {
        if (s < 0) throw new IllegalArgumentException("Negative salary");
        this.salary = s;
    }
    public void raise(double pct) { salary += salary * pct / 100; }
    public void raise(double pct, double bonus) { raise(pct); salary += bonus; }
    public static int getCount() { return count; }
    @Override public String toString() {
        return "Employee{id=" + id + ", name=" + name + ", salary=" + salary + "}";
    }
}
Extension challenges for fast finishers: (a) make name read-only after construction, (b) store employees in an ArrayList<Employee> and print with for-each (Day 8 tie-in), (c) add equals() comparing by id and test in a list.contains().
