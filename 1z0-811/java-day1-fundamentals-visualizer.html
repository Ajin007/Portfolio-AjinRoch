Section 1

☕ Java Introduction & History

Definition

Java is a class-based, object-oriented, platform-independent programming language designed to have as few implementation dependencies as possible. Code is compiled once into bytecode and executed anywhere a JVM exists.

Timeline

1991 — James Gosling & team at Sun Microsystems start the "Green Project"; language originally named Oak (later "Green", then Java).
1995 — Officially released as Java 1.0 with the promise "Write Once, Run Anywhere".
2010 — Oracle acquires Sun Microsystems; Java becomes an Oracle product.
2017 (Java 9+) — New 6-month release cadence; LTS (Long-Term Support) versions: 8, 11, 17, 21.
Certification questions love: creator = James Gosling, company = Sun Microsystems, original name = Oak, current owner = Oracle, LTS versions = 8 / 11 / 17 / 21.
WRONG: "Java is named after JavaScript."  →  RIGHT: Java and JavaScript are completely unrelated languages. Java was named after Java coffee.
Section 2

🌍 WORA — Write Once, Run Anywhere

Java source is compiled to bytecode (.class), not native machine code. Any OS with a JVM interprets/JIT-compiles that same bytecode — so one compiled artifact runs on Windows, Linux, and macOS unmodified.

Hello.java
Source code (human-readable)
→
javac
Compiler
→
Hello.class
Bytecode (platform-independent)
→
JVM (Win / Linux / Mac)
Executes on any OS
Bytecode is platform-independent; the JVM is platform-dependent. A separate JVM build exists for every OS — that's exactly what makes the bytecode portable.
WRONG: "Java compiles directly to machine code."  →  RIGHT: javac produces bytecode; the JVM (interpreter + JIT) converts it to machine code at runtime.

Section 4

🧩 JDK vs JRE vs JVM

JDK — Java Development Kit

JRE + development tools: javac, jar, javadoc, jdb, jshell

JRE — Java Runtime Environment

JVM + core class libraries (java.lang, java.util, …)

JVM — Java Virtual Machine

ClassLoader → Bytecode Verifier → Interpreter + JIT Compiler → Execution
JDKJREJVM
ContainsJRE + dev tools (javac, jar, javadoc)JVM + librariesExecution engine only
Who needs itDevelopers (write + compile)End users (run only)Abstract spec; ships inside JRE
Can compile?✅ Yes❌ No❌ No
PlatformPlatform-specific installerPlatform-specificPlatform-dependent implementation of a platform-independent spec
One-liner to remember: JDK = JRE + development tools; JRE = JVM + libraries. To only run a Java app you need the JRE; to develop you need the JDK.
Since Java 11, Oracle no longer ships a separate standalone JRE download — the JDK includes everything. Exam wording may still test the classical JDK/JRE/JVM containment model.
WRONG: "JVM converts source code to bytecode."  →  RIGHT: javac (in the JDK) makes bytecode; the JVM executes bytecode (interprets + JIT-compiles hot paths).

Section 7

✨ Java Features (Buzzwords)

Simple
C/C++-like syntax minus pointers, operator overloading & multiple class inheritance.
Object-Oriented
Everything lives inside classes; encapsulation, inheritance, polymorphism, abstraction.
Platform-Independent
Bytecode + JVM = WORA.
Secure
No explicit pointers; bytecode verifier; runs inside the JVM sandbox.
Robust
Strong type checking, exception handling, automatic garbage collection.
Multithreaded
Built-in Thread API; run tasks concurrently within one program.
Portable
Fixed primitive sizes on every platform (int is always 32-bit).
High Performance
JIT compiler converts hot bytecode into native machine code at runtime.
Distributed
Networking baked in (RMI, sockets, HTTP APIs).
Dynamic
Classes are loaded on demand at runtime (ClassLoader).
Interpreted + Compiled
Hybrid: bytecode is interpreted, hot spots are JIT-compiled.
Architecture-Neutral
Bytecode format is independent of CPU architecture.
WRONG: "Java is purely interpreted, so it's slow."  →  RIGHT: Modern JVMs use JIT compilation — hot code runs as native machine code.
"Java has no pointers" means no explicit pointer arithmetic for the programmer — references still exist internally.

Section 9

👋 HelloWorld — Anatomy of Your First Program

public class HelloWorld {                        // class name MUST match file name HelloWorld.java
    public static void main(String[] args) {      // JVM entry point — exact signature required
        System.out.println("Hello, World!");       // print with newline
    }
}
KeywordWhy it's required in main()
publicJVM must call it from outside the class — needs maximum visibility.
staticJVM calls it without creating an object of the class.
voidReturns nothing to the JVM.
mainThe exact method name the JVM looks for.
String[] argsCommand-line arguments. String... args (varargs) is also valid; the parameter name can be anything.
System.out.printlnSystem class → out static PrintStream field → println() method.
Valid main variations: static public void main(String[] a) ✅ (modifier order flexible), public static void main(String... args) ✅. Invalid as entry point: missing static, return type int, or misspelled Main.
Case sensitivity: Public, String[] Args vs args usage, and system.out (lowercase s) are classic Day-1 compile errors.
WRONG: "A file can contain only one class."  →  RIGHT: A file may contain multiple classes but only one public class, and its name must match the file name.

Common Day-1 errors

javac takes the file name (with .java); java takes the class name (no extension). Compile-time errors come from javac; runtime errors (exceptions) come from the JVM.
WRONG: "java HelloWorld.java always leaves a .class file."  →  RIGHT: Single-file source launch compiles in memory — no .class file is written to disk.
