Section 1

🏗️ Program Structure: package → import → class → main

The fixed order of a Java source file

package com.iamneo.training;                 // 1. package — OPTIONAL, max ONE, must be FIRST statement

import java.util.Scanner;                    // 2. imports — OPTIONAL, any number, after package
import java.util.ArrayList;

public class Demo {                          // 3. class declaration — file name must be Demo.java
    public static void main(String[] args) { // 4. main — JVM entry point
        System.out.println("Structure!");
    }
}
ElementRuleCompile error if violated?
packageAt most one; must be the very first non-comment statement✅ "class, interface, or enum expected"
importZero or more; between package and class✅ if placed after class
classMultiple allowed per file, but only ONE public; public class name = file name✅ name mismatch error
mainNeeded only for the class you launch with java❌ compiles fine; runtime error if missing when run
Mnemonic: P-I-C → Package, Import, Class — in that exact order. Comments may appear anywhere, even above package.
WRONG: "Every Java class needs a main method."  →  RIGHT: Only the class you launch needs main; libraries have thousands of classes without it.
Section 2

🏛️ OOP Pillars — 30,000-ft Overview

Today is only the fly-over — Day 9 lands the plane. Remember A-P-I-E:

Abstraction
Expose what an object does, hide how. Tools: abstract classes, interfaces. Real-world: you press the brake pedal — you don't operate brake fluid pressure.
Polymorphism
One interface, many forms. Compile-time = method overloading; runtime = method overriding.
Inheritance
A class acquires fields/methods of a parent via extends. Java allows only single class inheritance.
Encapsulation
Bundle data + methods; guard state with private fields + public getters/setters.
Exam phrasing map: "hiding internal implementation" → Abstraction · "data hiding / private + getters" → Encapsulation · "IS-A relationship" → Inheritance · "same method name, different behaviour" → Polymorphism.

Legal identifier rules (compiler-enforced)

Start with a letter, $ or _ (never a digit) · subsequent chars may include digits · no reserved keywords · case-sensitive · no length limit.

Conventions are not compiler rules — class employee compiles. Exams ask "which is legal" (compiler rules) vs "which follows convention" — read the question stem carefully.
Section 4

🔒 Reserved Keywords

Java reserves ~50 keywords you cannot use as identifiers. Grouped view:

GroupKeywords
Data typesbyte short int long float double char boolean
Flow controlif else switch case default for while do break continue return yield
Modifierspublic private protected static final abstract synchronized volatile transient native strictfp
Class-relatedclass interface enum record extends implements package import new this super instanceof
Exceptionstry catch finally throw throws assert
Unused (reserved)goto, const — reserved but do nothing; using them = compile error
true, false, null are literals, not keywords — but still illegal as identifiers. var is a contextual keyword (Java 10+): var x = 5; works and int var = 3; still compiles.
WRONG: "goto exists in Java like C."  →  RIGHT: goto is reserved but unusable; labeled break/continue cover its use-cases.

Section 6

📦 Packages & Imports

Why packages

Namespace management (two Employee classes can coexist), access control boundaries, and logical organisation mapped to folder structure.

Import forms

import java.util.Scanner;         // single-type import (preferred)
import java.util.*;               // wildcard — all classes in java.util (NOT subpackages)
import static java.lang.Math.max; // static import — use max(a,b) without Math.

java.util.Date d = new java.util.Date(); // fully-qualified name — no import needed
RuleDetail
java.lang is auto-importedString, System, Math, Integer… never need imports
Wildcard ≠ recursivejava.util.* does NOT include java.util.function.*
Wildcard has zero runtime costImports are compile-time only; no performance impact
Name clashjava.util.Date vs java.sql.Date both wildcard-imported → must fully qualify at use site
Same packageClasses in the same package never import each other
WRONG: "import loads the class into memory."  →  RIGHT: import is purely a compile-time naming shortcut; ClassLoader loads classes at runtime regardless.

Section 9

⌨️ Scanner — Console Input

import java.util.Scanner;

public class InputDemo {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        System.out.print("Age: ");
        int age = sc.nextInt();
        sc.nextLine();                        // consume the leftover newline!
        System.out.print("Name: ");
        String name = sc.nextLine();
        System.out.println(name + " is " + age);
        sc.close();
    }
}
MethodReadsNote
nextInt() / nextDouble()Next token as int/doubleThrows InputMismatchException on bad input
next()Single word (token)Stops at whitespace
nextLine()Rest of the line incl. spacesConsumes the newline
hasNextInt()Peek checkValidate before reading
The classic bug: nextInt() leaves the newline in the buffer, so a following nextLine() returns "" instantly. Fix: call an extra sc.nextLine() after numeric reads.
WRONG: next() reads a full sentence.  →  RIGHT: next() = one whitespace-delimited token; nextLine() = whole line.
