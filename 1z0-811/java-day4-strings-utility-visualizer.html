Section 1

🏊 String Pool & Immutability

Two ways to create — very different results

String a = "hello";              // literal → goes to STRING POOL (reused)
String b = "hello";              // SAME pooled object as a
String c = new String("hello"); // new → forced NEW object on the heap

a == b        // true  — same pool reference
a == c        // false — different objects
a.equals(c)   // true  — same characters
c.intern()    // returns the pooled "hello"

Immutability

String s = "Java";
s.concat(" SE 17");              // returns NEW string — s is UNCHANGED
System.out.println(s);           // "Java"
s = s.concat(" SE 17");          // must REASSIGN to keep the result

Why immutable? Pool sharing is only safe if nobody can mutate a shared string · thread-safety for free · stable hashCode → reliable HashMap keys · security (file paths, class names can't be tampered after checks).

Every String method that "modifies" (concat, replace, toUpperCase, trim, substring) actually returns a new String. If the result isn't assigned, it's silently discarded — a top-3 exam trick.
WRONG: "new String("hi") creates one object."  →  RIGHT: Potentially two — the pooled literal "hi" (if not already pooled) plus the new heap object.
Section 2

➕ Concatenation & StringBuilder

String full = "Java" + " " + 17;    // "Java 17" — + auto-converts any type
System.out.println(1 + 2 + "3");    // "33"  — left-to-right: 1+2=3, then "3"+"3"
System.out.println("1" + 2 + 3);    // "123" — once a String appears, all + = concat
System.out.println('A' + 1);        // 66 — char + int is ARITHMETIC, not concat!

Loops: use StringBuilder

// ❌ builds a new String object every pass — O(n²)
String s = "";
for (int i=0; i<1000; i++) s += i;

// ✅ mutable buffer — one object
StringBuilder sb = new StringBuilder();
for (int i=0; i<1000; i++) sb.append(i);
String out = sb.toString();

⚖️ String Comparison: equals() vs ==

ComparisonComparesUse for
==References (same object?)Primitives only; never String content
equals()Characters (same content?)The correct String comparison
equalsIgnoreCase()Content, case-insensitive"Java".equalsIgnoreCase("JAVA") → true
compareTo()Lexicographic order<0, 0, >0 — sorting; "apple".compareTo("banana") < 0
String a = "cert";
String b = new String("cert");
a == b               // false — different objects
a.equals(b)          // true  — same content

// null-safe pattern — constant first:
"admin".equals(userInput)     // never throws even if userInput is null
userInput.equals("admin")     // 💥 NPE when userInput is null
Two literals compared with == return true (pool!) — which makes buggy code "work" in demos and fail with runtime-built strings. Exams exploit exactly this: "ja"+"va" == "java" is true (compile-time constant folding), but s1+s2 == "java" with variables is false.
